01 — Introduction
What is MaCake?
Connecting bakers, clients & delivery drivers in one seamless ecosystem

MaCake is a mobile-first, multi-sided marketplace that connects passionate home bakers (Patissières) with customers who crave authentic, handcrafted cakes — and delivery drivers (Livreurs) who bring those sweet creations right to the door. Think of it as the Uber Eats of homemade pastries, with a unique InDrive-inspired pricing model for delivery negotiations.

Three Actors, One Platform

🛒
Client
Browse handmade cake catalogs, customize orders (colors, filling, message), pay securely via Stripe, and track delivery in real time.
  • Filter & search by category, price, rating
  • Fully personalized order customization
  • Live order status & real-time notifications
👩‍🍳
Patissière (Baker)
Showcase your creations, manage product listings, accept or refuse orders, and build a following of loyal customers.
  • Product catalog with photos, ingredients & pricing
  • Order acceptance & preparation workflow
  • Ratings & follower system to build reputation
🛵
Livreur (Delivery Driver)
Browse available delivery orders on an interactive map, propose a delivery price, and get paid upon completion.
  • Map-based workspace (OpenStreetMap)
  • Bid on delivery jobs with custom price proposals
  • Earnings tracked through wallet system

Key Algorithms & Patterns

🔑 JWT Token Refresh
Silent token rotation on 401 responses via Axios interceptors. Access tokens are short-lived; refresh tokens persist securely — balancing security and UX.
🤝 InDrive Estimation Algorithm
Delivery drivers submit price proposals for orders. The client or system confirms the best bid. This market-based pricing ensures fair compensation for drivers.
📡 Event-Driven Updates
RabbitMQ handles async events between services (e.g., order completion triggers rating prompt). WebSocket broadcasts real-time status changes to all connected clients.
⭐ Batch Rating Aggregation
The notation service computes average star ratings per product and per baker using batch queries — efficiently displayed in catalog listings without N+1 calls.

04 — Methodology
Strategy & Methodology
How we designed, built, and iterated MaCake

Development Approach

MaCake was built using an Agile-inspired iterative approach — starting with core domain modeling, then progressively layering services, integrations, and polish. Architecture decisions prioritized separation of concerns, scalability, and developer experience.

🎯
Domain-Driven Design
Each microservice owns exactly one bounded context: Auth owns identity, Catalog owns products, Orders owns the full transaction lifecycle. Clean boundaries prevent coupling.
🔄
API-First Development
All services were designed contract-first. The API Gateway enforces strict routing rules (role-based guards, JWT validation) before any request reaches a microservice.

Project Timeline

Phase 1 — Foundation
Architecture Design & Environment Setup
Defined microservice boundaries, set up Docker Compose with MongoDB, RabbitMQ, Redis, MinIO. Established gateway routing and TCP communication patterns.
Phase 2 — Core Services
Auth, Catalog & Order Services
Built user registration/login with JWT, product catalog CRUD with image uploads to MinIO, and the complete order lifecycle with status state machine.
Phase 3 — Platform Features
Payments, Estimations & Ratings
Integrated Stripe for secure payments in MAD currency, implemented the InDrive-style delivery price negotiation system, and built the ratings/likes/followers social layer.
Phase 4 — Frontend
React Native Mobile App
Built role-aware screens for all three user types using Expo Router, Redux Toolkit for state management, Lottie animations for delight, and OSM maps for delivery workspace.
Phase 5 — Polish & Optimization
Performance, Security & UX Refinement
Added rate limiting, solved rating edge cases, optimized error handling for payments, real-time WebSocket event delivery, and app-wide visual polish with NativeWind.

Infrastructure Strategy

🐳
Docker Compose
All services run in containers. One command to spin up the entire platform locally.
🗄️
DB per Service
Each microservice has its own isolated MongoDB database — no shared state, no data coupling.
📬
Message Queue
RabbitMQ decouples services. Events flow asynchronously — a service failure won't block others.
